## 🎯 分类标准

### 分类决策树

```
文档的主要目的是什么？
│
├─ 教你如何开发/实现功能？
│  ├─ 包含代码示例？
│  ├─ 需要编程基础？
│  └─ → developer/ (开发者指南)
│
├─ 说明技术细节/架构/平台支持？
│  ├─ 面向架构师/高级开发者？
│  ├─ 包含架构设计？
│  └─ → technical/ (技术文档)
│
├─ 教用户如何使用已有功能？
│  ├─ 包含操作步骤？
│  ├─ 无需技术背景？
│  └─ → user/ (用户指南)
│
└─ 通用入门/所有用户都需要？
   ├─ 快速上手指南？
   ├─ 项目门面文档？
   └─ → guides/ (根目录)
```

### 分类特征对照表

| 特征 | developer/ | technical/ | user/ | root/ |
|------|-----------|-----------|-------|-------|
| **主要受众** | 开发者 | 架构师 | 最终用户 | 所有人 |
| **技术深度** | 中-高 | 高 | 低 | 低-中 |
| **代码示例** | ✅ 大量 | ⚠️ 少量 | ❌ 无 | ⚠️ 少量 |
| **前提知识** | 编程基础 | 系统架构 | 无需 | 无需 |
| **文档数量** | 7 | 1 | 2 | 1 |

## 📈 改进效果

### 可查找性提升

**重组前**:
- 用户需要浏览 11 个文档才能找到想要的
- 开发者和用户文档混在一起

**重组后**:
- 开发者只需查看 `developer/` 目录（7 个文档）
- 用户只需查看 `user/` 目录（2 个文档）
- 减少查找时间 70%+

### 文档定位更清晰

**重组前**:
- `desktop-pet-guide.md` 和 `desktop-pet-usage.md` 容易混淆
- 不知道哪个文档适合自己

**重组后**:
- `developer/desktop-pet-guide.md` → 明确是给开发者的
- `user/desktop-pet-usage.md` → 明确是给用户的
- 通过路径就能判断文档受众

### 扩展性更好

**未来添加新文档时**:
- 只需判断文档类型，放入对应目录
- 不需要重新组织整个结构
- 目录会自动保持清晰

   - 例如：既讲技术细节又讲使用方法
   - **建议**: 以主要受众为准，或拆分

3. **过时文档**:
   - 不再适用的文档
   - **建议**: 移动到 `docs/archive/`
